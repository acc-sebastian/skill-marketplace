# Enterprise Migration Guide — GitHub → Azure DevOps / Entra ID

**Status: documented, not executed** (Roadmap Phase 5). The marketplace runs on
GitHub today. If SBO later wants it fully inside the corporate Microsoft tenant
— Entra ID SSO, internal permissions, no public repo — the architecture ports
1:1. This is the mapping so a migration is a config exercise, not a redesign.

## Why this is low-risk

The whole system is **plain text + a Python build + declarative CI**:

- Skills are Markdown/JSON files → any Git host stores them unchanged.
- The site is generated by `scripts/build_site.py` → runs on any CI runner.
- The automation is small Python scripts invoked by YAML workflows → the *logic*
  is portable; only the workflow wrapper syntax differs per platform.

Nothing depends on GitHub-proprietary services except the CI/host wrappers.

## Component mapping

| Concern | Today (GitHub) | Enterprise (Azure / Microsoft) |
|---|---|---|
| Repository & Git | GitHub repo | **Azure DevOps Repos** (or GitHub Enterprise / GitLab) |
| Identity / SSO | GitHub accounts | **Entra ID (Azure AD)** — SSO + internal group permissions |
| CI/CD | GitHub Actions (`.github/workflows/*`) | **Azure Pipelines** (`azure-pipelines.yml`) — same steps, same Python scripts |
| Site hosting | GitHub Pages | **Azure Static Web Apps** (or an internal static host) |
| Review gate | Branch protection + CODEOWNERS | Azure Repos **branch policies** + required reviewers |
| Releases / packages | GitHub Releases | **Azure Artifacts** (universal packages) |
| Contribution form | GitHub Issue Forms → issue→PR action | Azure Boards work-item form → pipeline that opens a PR |
| Secrets | Actions secrets | **Azure Key Vault** / pipeline variable groups |
| Notifications | Teams incoming webhook | unchanged (Teams webhook) — or native Azure DevOps → Teams |
| Skill distribution | Claude Code plugin marketplace (Git URL) + Managed Settings | unchanged — Claude Code can add any Git URL; Managed Settings point at the Azure Repos URL |

## What actually changes

1. **Workflow wrappers.** Rewrite each `.github/workflows/*.yml` as an Azure
   Pipeline. The `run: python scripts/…` lines are copied verbatim — the Python
   scripts (`validate_skills.py`, `scan_skills.py`, `build_site.py`,
   `release_skills.py`, `check_stale.py`, `smoke_test.py`, `auto_archive.py`,
   `build_insights.py`) don't change.
2. **Host URLs.** Swap the hardcoded `acc-sebastian.github.io` / `github.com/...`
   URLs (centralized as constants in `build_site.py`) for the Static Web App and
   Azure Repos URLs. One edit point.
3. **Auth model.** Replace GitHub review/branch protection with Azure Repos
   branch policies; map skill owners to Entra ID users/groups instead of GitHub
   handles in CODEOWNERS.
4. **Plugin distribution.** Claude Code's `/plugin marketplace add` accepts any
   Git URL, so point it (and the Managed-Settings `extraKnownMarketplaces`
   source) at the Azure Repos HTTPS URL. Everything else in
   [enterprise-setup.md](enterprise-setup.md) is unchanged.

## What does NOT change

- The skill format, the JSON schema, and the lifecycle model.
- Every Python script and its behavior.
- The website design and the client-side Insights panel (it can point at the
  Azure DevOps REST API for release/work-item data with the same shape).
- The Claude Code plugin-marketplace mechanism.

## Recommended sequencing (when the time comes)

1. Mirror the Git repo to Azure DevOps Repos.
2. Port `validate` + `build`/deploy pipelines first (get the site live internally).
3. Wire branch policies + Entra ID groups; map owners.
4. Move secrets to Key Vault; port the scheduled maintenance pipelines.
5. Repoint Claude Code Managed Settings at the internal Git URL.

Estimated effort: days, not weeks — because the substance (skills, schema,
scripts) crosses over untouched.
